Manejo de bases de datos con R

Brenda Georgina Tafoya Olvera

19/5/2020

RStudio ha trabajado en hacer lo más fácil posible el manejo de bases de datos en R, para esto ha trabajado en 2 aréas principales:

  • Conectar RStudio con fuentes de datos existentes como Spark, SQLServer, Oracle, Impala, Hive, etc, (Más info sobre esto aquí).

  • Uso de paqueterías dentro de RStudio como dplyr, DBI, odbc, etc.

Algunos conceptos…

Back-end database: database que es accesado por los usuarios de forma indirecta a través de una aplicación extrerna.

Query: consulta / lenguaje estándar que permite traer datos de una o más tablas, actualizar contenidos o eliminarlos, de una manera rápida.

ODBC: Open DataBase Conectivity; su objetivo es hacer posible el acceder a cualquier dato desde cualquier aplicación, sin importar qué sistema de gestión de bases de datos (DBMS) almacene los datos.

DBI

El paquete DBI Actúa como “intermediario” entre paquetes para permitir la conectividad con la base de datos del usuario u otros paquetes. Proporciona un conjunto coherente de funciones independientemente del tipo de base de datos a la que se accede.

El paquete define una interfaz implementada por los backends DBI como:

  • RMySQL
  • PostgreSQL
  • RSQLite
  • odbc
  • bigquery

Entre muchos otros.

DBI replica clases y métodos similares a los utilizados por diferentes DBMS como Java JDBC, Python DB-API y Microsoft ODBC.

Entre las opreaciones que podemos hacer con DBI tenemos:

  • conectar / deconectar al DBMS
  • crear y ejecutar estados en el DBMS
  • manejo de errores / excepciones
  • información de los objetos de la base de datos (metadatos)

Primeros pasos

Por lo general no se trabaja directamente con el paquete de DBI, se hace a través del database back-end que prefieras (algunos ejemplos mencionados arriba). Por lo que su instalación se hace autométicamente al instalar el database back-end.

Pero para este primer paso la instalación se puede llevar a cabo desde el CRAN como siempre:

install.packages("DBI")

Estructura de clase

Hay cuatro clases principales de DBI, tres de las cuales son extendidas por backends de bases de datos individuales:

  • DBIObject: Una clase base común para todos los DBI.
  • DBIDriver: una clase base que representa las propiedades generales de DBMS. Por lo general, las funciones del generador crean instancias de los objetos del controlador como RSQLite (), RPostgreSQL (), RMySQL (), etc.
  • DBIConnection: representa la conexión a una base de datos específica.
  • DBIResult: El resultado de un query o statement DBMS.

dbplyr

Además de trabajar con datos locales en memoria almacenados en dataframes, dplyr también funciona con datos remotos on-disc almacenados en bases de datos. Esto es particularmente útil en dos escenarios:

  • Los datos ya están en una base de datos.

  • Tenemos tantos datos que no caben todos en la memoria simultáneamente y necesita usar algún motor de almacenamiento externo (Si sus datos caben en la memoria, no tiene ninguna ventaja colocarlos en una base de datos; solo será más lento).

El uso más común de dbplyr se debe al primer caso.

Primeros pasos

Para usar bases de datos con dplyr, es necesario instalar dbplyr, esto NO es necesario si has instalado tidyverse con anterioridad.

install.packages("dbplyr")

El paquete dplyr depende del paquete DBI para la comunicación con las bases de datos. También será necesario instalar un paquete DBI backend. El paquete DBI proporciona una interfaz común que permite que dplyr trabaje con muchas bases de datos diferentes usando el mismo código. DBI se instala automáticamente con dbplyr, pero se debe instalar un backend específico para la base de datos a la que desea conectarse. Los más comunes son:

  • RMySQL
  • RSQLite
  • odbc
  • bigquery

En este caso usaremos RSQLite. SQLite es un motor de base de datos de dominio público, que implementa un subconjunto decente del SQL 92 estándar, incluidas las operaciones de creación, actualización, inserción y selección de tablas principales, además de la gestión de transacciones.

Instalamos…

install.packages("RSQLite")

Conectar a la base de datos

Para trabajar con una base de datos en dplyr, primero debemos conectarnos a ella utilizando DBI :: dbConnect ().

  • Los argumentos que recibe varían dependiendo de la base de datos, pero el primer argumento es siempre el database backend que vayamos a utilizar, en nuestro caso se vería algo así:
library(dplyr)
con <- DBI::dbConnect(RSQLite::SQLite(), path = ":dbname:")
  • La mayoría de las bases de datos existentes no viven en un archivo, sino que viven en otro servidor. En la vida real, el código se verá más o menos así:
con <- DBI::dbConnect(RMySQL::MySQL(), 
  host = "database.rstudio.com",
  user = "hadley",
  password = rstudioapi::askForPassword("Database password")
)

Generar queries / ejemplo

Para interactuar con bases de datos generalmente se utiliza SQL (Structured Query Language).

  • La meta de dbplyr es generar SQL de forma automática para que tu no debas, es decir, poder usar la mayoría de los statements que ofrece SQL sin la necesidad de ser experto en SQL.

  • Como consecuencia, dbplyr no tiene la misma capacidad que ofrece el lenguaje de SQL pues se enfoca en los statements del tipo SELECT, ya que son los más usados.

Veamos un ejemplo de cómo se mezclan dplyr y SQL, para esto copiaremos los datos del dataset nycflights13, usando la función copy_to() a nuestra database creada arriba.

También configuramos índices que nos permitirán procesar rápidamente los datos por día, transportista, avión y destino.

copy_to(con, nycflights13::flights, "flights",
  temporary = FALSE, 
  indexes = list(
    c("year", "month", "day"), 
    "carrier", 
    "tailnum",
    "dest"
  )
)

Podemos llevar a cabo de forma eficiente algunas de las funciones básicas que ofrece SQL sin necesidad de aprender la gramática de SQL.

Observaciones:

  • La diferencia más importante entre los dataframes ordinarios y las consultas de bases de datos remotas es que su código R se traduce a SQL y se ejecuta en la base de datos, no en R.

    • Nunca extrae datos en R a menos que lo solicite explícitamente.

  • Al trabajar con bases de datos, dplyr intenta ser lo más vago posible, es decir, los queries que crea no son exactamente cómo se harían en SQL.

    • Retrasa cualquier trabajo hasta el último momento posible: reúne todo lo que quremos hacer y luego lo envía a la base de datos en un solo paso.

Ahora, creamos una tabla con la función dbWriteTable() la cuál contiene 3 argumentos básicos:

  • conn: conexión con la SQLite database
  • name: nombre que le asignamos a la tabla
  • value: los datos que queremos insertar (por lo general un dataframe)
# creamos la tabla mtcars_data
dbWriteTable(conn, "cars_data", mtcars)

# nos aseguramos de haber creado correctamente la tabla
dbListTables(conn)
