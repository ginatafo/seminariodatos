Algunos conceptos

Back-end database: database que es accesado por los usuarios de forma indirecta a travs de una aplicacin extrerna.

Query: consulta / lenguaje estndar que permite traer datos de una o ms tablas, actualizar contenidos o eliminarlos, de una manera rpida.

ODBC: Open DataBase Conectivity; su objetivo es hacer posible el acceder a cualquier dato desde cualquier aplicacin, sin importar qu sistema de gestin de bases de datos (DBMS) almacene los datos.

DBI

El paquete DBI Acta como intermediario entre paquetes para permitir la conectividad con la base de datos del usuario u otros paquetes. Proporciona un conjunto coherente de funciones independientemente del tipo de base de datos a la que se accede.

El paquete define una interfaz implementada por los backends DBI como:

  • RMySQL
  • PostgreSQL
  • RSQLite
  • odbc
  • bigquery

Entre muchos otros.

DBI replica clases y mtodos similares a los utilizados por diferentes DBMS como Java JDBC, Python DB-API y Microsoft ODBC.

Entre las opreaciones que podemos hacer con DBI tenemos:

  • conectar / deconectar al DBMS
  • crear y ejecutar estados en el DBMS
  • manejo de errores / excepciones
  • informacin de los objetos de la base de datos (metadatos)

Primeros pasos

Por lo general no se trabaja directamente con el paquete de DBI, se hace a travs del database back-end que prefieras (algunos ejemplos mencionados arriba). Por lo que su instalacin se hace automticamente al instalar el database back-end.

Pero para este primer paso la instalacin se puede llevar a cabo desde el CRAN como siempre:

install.packages("DBI")

Estructura de clase

Hay cuatro clases principales de DBI, tres de las cuales son extendidas por backends de bases de datos individuales:

  • DBIObject: Una clase base comn para todos los DBI.
  • DBIDriver: una clase base que representa las propiedades generales de DBMS. Por lo general, las funciones del generador crean instancias de los objetos del controlador como RSQLite (), RPostgreSQL (), RMySQL (), etc.
  • DBIConnection: representa la conexin a una base de datos especfica.
  • DBIResult: El resultado de un query o statement DBMS.

dbplyr

Adems de trabajar con datos locales en memoria almacenados en dataframes, dplyr tambin funciona con datos remotos on-disc almacenados en bases de datos. Esto es particularmente til en dos escenarios:

  • Los datos ya estn en una base de datos.

  • Tenemos tantos datos que no caben todos en la memoria simultneamente y necesita usar algn motor de almacenamiento externo (Si sus datos caben en la memoria, no tiene ninguna ventaja colocarlos en una base de datos; solo ser ms lento).

El uso ms comn de dbplyr se debe al primer caso.

Primeros pasos

Para usar bases de datos con dplyr, es necesario instalar dbplyr, esto NO es necesario si has instalado tidyverse con anterioridad.

install.packages("dbplyr")

El paquete dplyr depende del paquete DBI para la comunicacin con las bases de datos. Tambin ser necesario instalar un paquete DBI backend. El paquete DBI proporciona una interfaz comn que permite que dplyr trabaje con muchas bases de datos diferentes usando el mismo cdigo. DBI se instala automticamente con dbplyr, pero se debe instalar un backend especfico para la base de datos a la que desea conectarse. Los ms comunes son:

  • RMySQL
  • RSQLite
  • odbc
  • bigquery

En este caso usaremos RSQLite. SQLite es un motor de base de datos de dominio pblico, que implementa un subconjunto decente del SQL 92 estndar, incluidas las operaciones de creacin, actualizacin, insercin y seleccin de tablas principales, adems de la gestin de transacciones.

Instalamos

install.packages("RSQLite")

Conectar a la base de datos

Para trabajar con una base de datos en dplyr, primero debemos conectarnos a ella utilizando DBI :: dbConnect ().

  • Los argumentos que recibe varan dependiendo de la base de datos, pero el primer argumento es siempre el database backend que vayamos a utilizar, en nuestro caso se vera algo as:
library(dplyr)
con <- DBI::dbConnect(RSQLite::SQLite(), path = ":dbname:")
  • La mayora de las bases de datos existentes no viven en un archivo, sino que viven en otro servidor. En la vida real, el cdigo se ver ms o menos as:
con <- DBI::dbConnect(RMySQL::MySQL(), 
  host = "database.rstudio.com",
  user = "hadley",
  password = rstudioapi::askForPassword("Database password")
)

Generar queries / ejemplo

Para interactuar con bases de datos generalmente se utiliza SQL (Structured Query Language).

  • La meta de dbplyr es generar SQL de forma automtica para que tu no debas, es decir, poder usar la mayora de los statements que ofrece SQL sin la necesidad de ser experto en SQL.

  • Como consecuencia, dbplyr no tiene la misma capacidad que ofrece el lenguaje de SQL pues se enfoca en los statements del tipo SELECT, ya que son los ms usados.

Observaciones:

  • La diferencia ms importante entre los dataframes ordinarios y las consultas de bases de datos remotas es que su cdigo R se traduce a SQL y se ejecuta en la base de datos, no en R.

    • Nunca extrae datos en R a menos que lo solicite explcitamente.

  • Al trabajar con bases de datos, dplyr intenta ser lo ms vago posible, es decir, los queries que crea no son exactamente cmo se haran en SQL.

    • Retrasa cualquier trabajo hasta el ltimo momento posible: rene todo lo que quremos hacer y luego lo enva a la base de datos en un solo paso.
